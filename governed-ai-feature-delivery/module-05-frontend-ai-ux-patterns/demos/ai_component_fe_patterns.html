idle / queued
Before the request
Show your output skeleton — mirroring the actual field layout, not a generic spinner. Queued means the server has the request but hasn't started generating. Surface this if latency will be noticeable (>1s).
streaming / partial
Tokens arriving
Render tokens as they arrive with no layout shift. Use min-height reservation. Never attach status badges, confidence labels, or commit actions to partial output. The stream is not the result.
complete
Stream closed cleanly
Transition field statuses from skeleton to real. Enable the review step. Surface confidence bands and evidence. The confirmation gesture and audit write happen here — not before.
needs_review
Backend flagged for human check
This is a defined state, not an error. Render it differently: highlight the uncertain fields, surface a "review required" header, require explicit sign-off before commit. Never silently treat this as complete.
SSE setup
Use ReadableStream + TextDecoder
Avoid raw string concatenation on the response body. Use getReader() on the response body, decode with TextDecoder, and parse SSE data: lines explicitly. Handle [DONE] as a first-class event.
Layout stability
Reserve space before tokens arrive
Set a min-height on the output container that matches your expected output height. Otherwise each token causes a reflow. For structured field output, render all field rows immediately in skeleton state, then populate values as they stream in.
Interruption
Define the broken stream state
A stream that stops mid-output is not complete. Store the partial result separately from the final result. Offer a retry. Never display partial output with final-output styling or make it commitable.
Progress signalling
Show deterministic stages, not a spinner
Users tolerate waiting better when they can see progress. Use stage labels (queued → extracting → reviewing → done) rather than an indeterminate spinner. Tie labels to actual backend events from your SSE stream where possible.

Each error type has a different recovery action. Generic error messages hide which recovery is appropriate.

network Request never reached the server or timed out in transit. Recovery: automatic retry with exponential backoff, then "Try again" CTA. Preserve the input — don't clear the form.
model_error Server returned an error from the model or backend pipeline. Surface the error_type from your API contract. Recovery action depends on type — don't collapse all model errors to one message.
timeout Stream started but didn't complete in the expected window. Distinguish from network failure — the partial output may be useful. Offer retry with reduced scope (fewer fields, shorter input) as a secondary action.
policy_blocked Not an error — a policy decision. Render with a distinct colour (purple / info, not red). Explain what was blocked and why at field level, not just page level. The CTA is "request review" not "retry".
partial_failure Some fields succeeded, one or more failed. Never show a single error state for the whole form — show per-field status. The user can still review and commit the fields that did succeed, with the failed ones flagged.
stream_interrupted SSE connection closed before [DONE]. Store partial as a separate state. Show "Incomplete result — retry?" rather than discarding or presenting as final.
Do
  • Define all states before writing any render logic
  • Match your UI state enum to your backend response contract
  • Show field-level status, not page-level confidence
  • Require explicit confirmation before any high-impact commit
  • Reserve layout space before tokens arrive
  • Make uncertain fields visually prominent, not subtle
  • Persist the reviewed state and confirmation gesture for audit
  • Treat needs_review as a first-class state
  • Give each error type its own recovery path
  • Keep display components pure — review wrapper owns edit state
Avoid
  • One-click commit on unreviewed model output
  • Generic spinner instead of output skeleton
  • Styling partial output as final
  • A single confidence score with no band definition
  • Hiding uncertainty to make the UI look cleaner
  • Conflating network error and model error in one message
  • UI state enum that diverges from backend response statuses
  • Inconsistent fallback behaviour across screens
  • Layout shift as tokens arrive
  • Exposing internal prompt text or policy internals to users
