# Final Build and Review

**Module 8 — Governed AI Feature Delivery**

---

## Build objective

Deliver a production-style AI feature with governance built in.

- <span class="fragment">Backend workflow and validation layers</span>
- <span class="fragment">Bounded tool usage where needed</span>
- <span class="fragment">Structured frontend review experience</span>
- <span class="fragment">Evaluation results and release criteria</span>

---

## Why this final module matters

- <span class="fragment">Isolated module knowledge is not enough for production delivery.</span>
- <span class="fragment">Integration exposes real trade-offs and weak boundaries.</span>
- <span class="fragment">Teams need evidence-backed decisions, not one-time demos.</span>
- <span class="fragment">This is where reusable delivery patterns are proven.</span>

---

## Final build scope

Your team solution must include:

- <span class="fragment">Backend workflow with clear orchestration boundaries</span>
- <span class="fragment">Guardrails for input, tool, and output paths</span>
- <span class="fragment">Frontend review flow with uncertainty handling</span>
- <span class="fragment">Evaluation suite and quality results</span>
- <span class="fragment">Deployment and governance readiness pack</span>

---

## Integration architecture checklist

- <span class="fragment">Controller, workflow, gateway responsibilities are clear</span>
- <span class="fragment">Prompt/model versions are explicit and traceable</span>
- <span class="fragment">Validation and fallback paths remain deterministic</span>
- <span class="fragment">UI states align with backend status contract</span>

---

## Evidence required for review

| Evidence type | Example artifacts |
| ------------- | ----------------- |
| Architecture | component flow + boundary map |
| Quality | eval pass/fail summary + dataset version |
| Safety | guardrail rule set + fallback examples |
| Operations | release gate checklist + rollback/fallback plan |
| Observability | trace samples with provenance fields |

---

## Review rubric

- <span class="fragment">Correctness and schema reliability</span>
- <span class="fragment">Safety and trust boundary handling</span>
- <span class="fragment">UX clarity for uncertainty and evidence</span>
- <span class="fragment">Observability and deployment readiness</span>

---

## Demo expectations

1. <span class="fragment">Show one end-to-end request lifecycle.</span>
2. <span class="fragment">Show one failure path and fallback behavior.</span>
3. <span class="fragment">Show eval evidence used for release decision.</span>
4. <span class="fragment">Show trace evidence for auditability.</span>
5. <span class="fragment">State go/no-go recommendation and rationale.</span>

---

## Common final-build failure modes

- <span class="fragment">Feature works once, but lacks repeatable evidence.</span>
- <span class="fragment">Strong backend but unclear frontend review behavior.</span>
- <span class="fragment">Good eval scores but no operational rollout plan.</span>
- <span class="fragment">Trace data present but not decision-useful.</span>
